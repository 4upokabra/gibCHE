# ТЗ Dev A — Core & интеграции

## Цель
Спроектировать и реализовать ядро сбора разведданных: архитектуру модулей, подключение внешних сканеров, нормализацию данных и их подготовку для дальнейшей аналитики и атакующих модулей.

## Основные задачи
1. Описать архитектуру сервисов (`scanner-core`, `attack-engine`, `analysis-LLM`, `ui-gateway`) и схему взаимодействия между ними.
2. Развернуть базу данных (PostgreSQL/ClickHouse) и очередь задач (Redis + RQ/Celery), подготовить Docker-компоуз для локальных стендов.
3. Реализовать обёртки для `nmap`, `Shodan`, `VirusTotal`, пассивного DNS/WHOIS и других источников, предусмотрев единую схему конфигурации.
4. Построить сервис агрегации результатов сканов: нормализация, дедупликация, enrichment метаданными (гео, принадлежность, техстек).
5. Спроектировать и задокументировать формат событий и хранилище историй сканирования (включая статус выполнения и ошибки).
6. Настроить базовый мониторинг (Prometheus/Grafana/Logs) и протокол логирования для дальнейшей интеграции с backend-приложением.

## API и интерфейсы
- REST/gRPC эндпоинты для планирования сканов, получения статуса задания и выгрузки результатов.
- Формат сообщений в очереди: `TaskEnvelope` (источник, параметры, SLA, тип модели тестирования).
- Контракты сериализации для передачи данных в LLM-модуль и backend (Dev D).

## Требования к данным и инфраструктуре
- Все результаты сканирования должны храниться с указанием источника, времени и уровня доверия.
- Предусмотреть idempotency ключи для повторного запуска сканов.
- Секреты внешних API хранить через переменные окружения `.env` + шаблон `.env.example`.

## Критерии готовности
- Поднята инфраструктура (БД, очередь, сервисы) в Docker и задокументирован запуск.
- Минимум три внешних сканера подключены и результаты доступны через API.
- Есть схемы данных (SQL/ER диаграмма) и Swagger/Proto спецификация для публичных методов.
- Настроены базовые метрики и алерты по критическим ошибкам сканирования.
